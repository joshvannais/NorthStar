# M19.5 Phase B — Polaris Fact Model Specification

**Version:** v1.0  
**Status:** Owner-ratified specification — no code changes until Phase C approval  
**Author:** NorthStar Solutions Team  
**Date:** 2026-07-20

---

## 1. Executive Recommendation

The current Polaris intelligence pipeline produces structurally valid output built on fundamentally unsound evidence. The canonical record exists, but its facts are lossy, speaker-blind, and provenance-free. The fix is a typed fact model with strict evidence rules — not more keyword patches.

**Recommendation:** Adopt the schema defined below. It is the smallest model that:

- Preserves speaker boundaries and sentence-level evidence
- Distinguishes customer-stated from agent-sourced or inferred facts
- Supports multi-entity jobs (tree groups, HVAC systems, roof sections)
- Carries a typed, normalized value separate from display text
- Uses `extractionConfidence` to measure extraction quality — not estimate confidence, verification confidence, or qualification completeness
- Enables deterministic qualification, estimate eligibility, and actionability

The model is designed to coexist with the current flat `polarisIntelligence` record during a staged migration. No existing consumer breaks in Phase C.

## 1a. Owner-Ratified Decisions (2026-07-20)

The following decisions were made by the owner during Phase B review and are incorporated into this specification. They are binding for Phases C–F.

### Decision 1: Confidence Field

Use `extractionConfidence` — not a generic `confidence` field.

**Definition:** `extractionConfidence` measures how confident Polaris is that the normalized fact was correctly extracted from the cited evidence.

It must NOT represent:
- Estimate confidence
- Lead confidence
- Customer-intent confidence
- Verification confidence
- Qualification completeness
- Confidence in an inferred conclusion

Do not add `verificationConfidence` yet. Future verification confidence may be introduced only when NorthStar has a real verification mechanism (repeated customer confirmation, cross-source agreement, business records, property data, completed-job outcomes, or human review).

**Rule:** A high `extractionConfidence` must not override an invalid status. Example: `status: mentioned_unresolved` with `extractionConfidence: 0.92` still does not count as `collected` and cannot directly affect pricing.

### Decision 2: Tree Entity Model

Approved as specified: `treeGroups[]` containing nested `trees[]` where individual separation is possible. The model must support one tree, multiple similar trees described as a group, multiple trees with different facts, group-level facts, per-tree facts, and site-level facts. Do not force artificial individual tree records when the customer only provides grouped information.

### Decision 3: Conflict Resolution

Keep conflicting facts rather than silently applying last-statement-wins. Preserve every conflicting customer-supported value. Mark the canonical fact as `conflicting`. Retain evidence for each candidate value. Do not count the fact toward completeness. Do not allow it to affect pricing. Generate a clarification action.

The latest statement may be identified as the most recent candidate, but must not automatically replace earlier evidence unless the customer explicitly corrects the prior statement. Explicit correction ("Actually, I was wrong") may resolve to the corrected value when the correction relationship is clearly detected and both evidence turns remain preserved.

### Decision 4: Inferred Facts

Approved for uncertainty/range-widening only. Inferred facts must NOT increase an estimate multiplier, count toward qualification completeness, appear as confirmed customer facts, receive a green `collected` indicator, be represented as directly stated, or resolve missing required information. They may widen the opportunity range, lower estimate confidence, trigger a follow-up question, require an on-site evaluation, or appear in reasoning as a clearly labeled assumption.

### Decision 5: Fixture Storage

Approved location: `tests/fixtures/polaris/` with subdirectories for transcripts, expected-facts, expected-intelligence, retell-payloads, and simulation-payloads. The failed production-shaped Tree Service scenario must be included as a permanent regression fixture.

## 2. Canonical Fact Schema

### 2.1 Core Fact Record

```typescript
interface PolarisFact {
  // ── Identity ──
  id: string;                    // UUID v4, generated at extraction time
  entityType: string;            // 'job' | 'customer' | 'site' | 'tree' | 'tree_group' | 'hvac_system' | 'room' | 'roof_section' | etc.
  entityId: string;              // UUID or index within the parent entity (e.g. treeGroup-0, treeGroup-1)
  variable: string;              // e.g. 'tree_height', 'property_sqft', 'urgency', 'service_address'

  // ── Value ──
  valueType: string;             // 'measurement' | 'quantity' | 'age' | 'duration' | 'boolean' | 'category' | 'free_text' | 'address' | 'phone' | 'email' | 'datetime' | 'hazard' | 'entity_ref'
  status: FactStatus;            // see Section 3
  normalizedValue: number | boolean | string | null;  // machine-usable value (e.g. 120, true, "emergency")
  displayValue: string | null;   // human-readable form (e.g. "Approximately 120 ft", "High urgency")
  unit: string | null;           // 'ft', 'in', 'sqft', 'yr', 'min', '$', null for non-measurements

  // ── Extraction Confidence ──
  extractionConfidence: number;  // 0.0–1.0. Measures how confidently the fact was extracted from cited evidence.
                                 // 0.9+ = directly stated, unambiguous, customer-sourced
                                 // 0.7+ = clearly stated with minor ambiguity
                                 // 0.4+ = implied or partially stated
                                 // <0.4 = weak inference, not usable for estimation
                                 // Does NOT represent estimate confidence, lead confidence,
                                 // verification confidence, or qualification completeness.

  // ── Evidence ──
  evidence: FactEvidence;

  // ── Provenance ──
  extractionMethod: string;      // 'direct_extraction' | 'speaker_inference' | 'system_inference' | 'webhook' | 'legacy_migration'
  provenance: string;            // Short description of how this fact was determined
  conflicts: string[];           // IDs of PolarisFact records that contradict this one

  // ── Timestamps ──
  createdAt: string;             // ISO 8601
  updatedAt: string;             // ISO 8601, same as createdAt on first creation
}
```

### 2.2 Evidence Record

```typescript
interface FactEvidence {
  speaker: 'customer' | 'agent' | 'system' | 'unknown';
  turnId: string | null;         // Transcript turn identifier (e.g. "turn-3", "sim-msg-7")
  utterance: string | null;      // Full utterance text from the transcript
  exactSpan: string | null;      // The exact substring that produced the fact (e.g. "one hundred and twenty feet")
  timestamp: string | null;      // ISO 8601 or relative offset
  transcriptSource: 'retell' | 'simulation' | 'webhook' | 'manual';
}
```

### 2.3 Design Decisions

- **`entityType` + `entityId`** — This is the smallest way to support multi-entity jobs. A Tree Service job produces multiple `PolarisFact` records with `entityType: 'tree'` and `entityId: '0'`, `'1'`, etc. A single-tree job uses `entityId: '0'` or a single `treeGroup` with `quantity: 1`.
- **`normalizedValue`** — Always the machine-usable form. `null` when the fact was mentioned but no value was extracted (status `mentioned_unresolved`).
- **`displayValue`** — Always a human-readable string or null. Never invented — must trace to evidence.
- **`extractionConfidence`** — Extraction quality, not "how sure are we this is true." A clean customer statement like "the tree is about 80 feet tall" gets 0.9. A guessed value from a keyword match with no direct customer statement gets 0.2. Does NOT represent estimate confidence, lead confidence, or verification confidence.
- **`evidence`** — Required for all statuses except `missing` and `not_applicable`. `system` speaker is used for facts inferred from non-transcript sources (e.g. business profile defaults, industry baselines).
- **`conflicts`** — An array of fact IDs, not a boolean. This allows multiple conflicting facts to coexist (e.g. customer says "80 feet" then later "maybe 100 feet").

## 3. Fact Status Invariants

### 3.1 Status Definitions

| Status | Definition | Normalized Value Required | Customer Evidence Required | Counts Toward Completeness | May Affect Estimate | Green Check in UI | May Appear in Briefing | May Generate Action |
|---|---|---|---|---|---|---|---|---|
| `collected` | Customer directly stated a value. Clearly extracted and validated. | Yes | Yes | Yes | Yes | Yes | Yes | No (already collected) |
| `mentioned_unresolved` | Customer mentioned the variable but no specific value was extracted. | No | Yes | No | No (increases uncertainty) | No | Yes (as "mentioned but not specified") | Yes ("Confirm the [variable]") |
| `missing` | Variable was not mentioned at all. | No | No | No | No (increases uncertainty) | No | Yes (as "not discussed") | Yes ("Ask about [variable]") |
| `conflicting` | Multiple customer statements disagree, or customer statement contradicts system data. | No | Yes | No | No (flags for review) | No | Yes (as "conflicting information") | Yes ("Verify [variable]") |
| `inferred` | Derived from other evidence (e.g. property size from address lookup, urgency from keywords). | Yes | No | No | Only if explicitly permitted (see Section 8) | No (uses distinct icon) | Yes (as "inferred from [source]") | Yes ("Confirm inferred [variable]") |
| `not_applicable` | Variable does not apply to this job (e.g. stump removal for a pruning job). | No | No | N/A | No | N/A | No | No |

### 3.2 Invariant Rules

1. **`collected` is the only status that counts toward qualification completeness.** A variable must have status `collected` with a non-null `normalizedValue` and `evidence.speaker === 'customer'` to count as qualified.

2. **`collected` is the only status that directly affects pricing.** Inferred facts may affect the estimate range width or confidence, but not the base multiplier.

3. **`conflicting` facts are shown as unresolved.** The UI must display them with a warning icon and the customer's conflicting statements. Do not average or pick one silently.

4. **`mentioned_unresolved` is distinct from `missing`.** The customer engaged with the topic but didn't provide a value. This is stronger signal than `missing` and should generate a specific follow-up question.

5. **`inferred` facts must be labeled.** Any brief or estimate that uses an inferred fact must state "Inferred from [source] — not customer-stated."

6. **Agent speech is never `collected`.** A fact sourced from `evidence.speaker === 'agent'` is automatically `inferred` at best, regardless of the extraction mechanism.

## 4. Speaker and Evidence Rules

### 4.1 Transcript Normalization

Both Retell and simulation transcripts produce a common turn format:

```typescript
interface TranscriptTurn {
  turnId: string;
  speaker: 'customer' | 'agent' | 'system';
  text: string;
  timestamp: string;          // ISO 8601 or relative offset
  source: 'retell' | 'simulation';
}
```

Normalization maps:
- Retell `role: 'user'` → `speaker: 'customer'`
- Retell `role: 'assistant'` → `speaker: 'agent'`
- Simulation `from: 'You'` → `speaker: 'agent'`
- Simulation `from: 'Customer'` → `speaker: 'customer'`

### 4.2 Evidence Isolation Rules

1. **Service detection** — Only customer utterances may determine the requested service. Agent confirmation questions ("So you need tree removal?") are not evidence.

2. **Urgency detection** — Only customer-stated urgency keywords count. Agent questions ("Is this an emergency?") must not contribute. The word "emergency" in an agent question is a question, not a statement.

3. **Measurement extraction** — Only customer utterances containing numbers + measurement units are candidates. Agent repetition of customer numbers ("So 80 feet tall?") is not independent evidence.

4. **Qualification** — Only customer-sourced `collected` facts count toward the qualification score.

5. **Estimate adjustments** — Only customer-sourced `collected` facts may apply multipliers. Agent-sourced or inferred facts may widen the range but not shift the midpoint.

### 4.3 System Inferences

Some facts are legitimately inferred from non-transcript sources:

- **Business profile** — Industry, service area, standard pricing
- **Call metadata** — Duration, call time, caller number
- **Address lookup** — Property size, zone, access notes
- **Historical data** — Repeat customer, past job types

These are represented as `inferred` with `evidence.speaker: 'system'` and a clear `provenance` string. They are never `collected`.

## 5. Typed Values

### 5.1 Value Type Definitions

| Value Type | normalizedValue Type | Example | Unit | Display |
|---|---|---|---|---|
| `measurement` | `number` | `120` | `'ft'` | "120 ft" |
| `quantity` | `number` | `4` | `'trees'` | "4 trees" |
| `age` | `number` | `10` | `'yr'` | "10 years old" |
| `duration` | `number` | `15` | `'min'` | "15 minutes" |
| `boolean` | `boolean` | `true` | `null` | "Yes" / "No" |
| `category` | `string` | `'emergency'` | `null` | "Emergency" |
| `free_text` | `string` | `'Large oak near the house'` | `null` | raw text |
| `address` | `string` | `'123 Main St'` | `null` | formatted address |
| `phone` | `string` | `'+15551234567'` | `null` | "(555) 123-4567" |
| `email` | `string` | `'a@b.com'` | `null` | raw email |
| `datetime` | `string` (ISO 8601) | `'2026-07-22T09:00:00Z'` | `null` | "Thursday, July 22 at 9:00 AM" |
| `hazard` | `string` | `'near_power_lines'` | `null` | "Near power lines" |
| `entity_ref` | `string` | `'tree_group-0'` | `null` | references another entity |

### 5.2 Normalized vs Display vs Original

```
Customer: "about one hundred and twenty feet"

  normalizedValue: 120
  unit:            'ft'
  displayValue:    "Approximately 120 ft"
  evidence.exactSpan: "one hundred and twenty feet"
```

- **`normalizedValue`** — Machine-usable. Always parsed from the evidence. `120` (number), not `"120"` or `"one hundred and twenty"`.
- **`displayValue`** — Human-readable. May include qualifiers like "approximately", "about", "over". Never adds information not present in the evidence.
- **`evidence.exactSpan`** — The raw substring from the transcript. Preserves the original wording.

### 5.3 Number Parsing Rules

The parser must handle:
- Simple digits: `"80"` → 80
- Hyphenated: `"eighty-foot"` → 80
- Spoken compound: `"one hundred and twenty"` → 120
- Mixed: `"four 120-foot trees"` → quantity=4, height=120
- Range: `"80 to 100 feet"` → two facts (height_min=80, height_max=100) or a single `measurement` with `conflicting` status
- Approximate: `"about 50 feet"` → 50 with lower `extractionConfidence`
- Comparison: `"taller than the house"` → `mentioned_unresolved` (no value)

## 6. Entity Model

### 6.1 Entity Hierarchy

```
Lead
 └── Customer (name, phone, email, address, preferredContact)
 └── Job (industry, requestedService, urgency, siteAccess)
      ├── Site Facts (propertySize, accessNotes, hazards)
      ├── Asset Groups (treeGroups[], hvacSystems[], rooms[], roofSections[])
      │    └── Individual Assets (tree[], hvacUnit[], room[], section[])
      └── Job-Level Facts (duration, callOutcome, requiresSiteEvaluation)
```

### 6.2 Tree Service Entity Model

```typescript
interface TreeGroup {
  entityType: 'tree_group';
  entityId: string;              // 'tree_group-0', 'tree_group-1'
  quantity: PolarisFact;         // "4 trees", normalizedValue: 4
  species: PolarisFact;          // "Pine", category
  trees: Tree[];                 // Individual tree records

  // Group-level facts
  location: PolarisFact;         // "Backyard", "Near house"
  accessDifficulty: PolarisFact; // "Easy", "Moderate", "Difficult"
  equipmentNeeded: PolarisFact;  // "Bucket truck", "Crane"
  debrisRemoval: PolarisFact;    // boolean
}

interface Tree {
  entityType: 'tree';
  entityId: string;              // 'tree-0', 'tree-1' within the group
  height: PolarisFact;
  trunkDiameter: PolarisFact;
  condition: PolarisFact;        // "Healthy", "Dead", "Diseased", "Storm damaged"
  leanDirection: PolarisFact;    // "Toward house", "Away from structures"
  hazards: PolarisFact[];        // "Near power lines", "Overhanging roof", "Root damage"
  removalRequired: PolarisFact;  // boolean
}

// Job-level facts
interface TreeServiceJobFacts {
  stumpRemoval: PolarisFact;     // boolean
  stumpGrinding: PolarisFact;    // boolean
  siteAccess: PolarisFact;       // "Easy", "Moderate", "Difficult"
  permitRequired: PolarisFact;   // boolean
  disposalMethod: PolarisFact;   // "Haul away", "Leave on site", "Split"
}
```

### 6.3 Other Industry Entity Models (Preview)

The same pattern extends to other industries:

**HVAC:**
```
hvacSystems[]
  └── hvacUnit[]
       ├── type (furnace, AC, heat pump, ductless)
       ├── age
       ├── fuelType
       ├── capacity (tons/BTUs)
       ├── condition
       └── symptoms[]
```

**Painting:**
```
rooms[]
  └── room[]
       ├── dimensions (sqft)
       ├── ceilingHeight
       ├── surfaceType (drywall, wood, brick)
       ├── prepWork (patching, sanding, priming)
       └── paintType (interior, exterior, primer, stain)
```

**Roofing:**
```
roofSections[]
  └── section[]
       ├── pitch
       ├── squares
       ├── layers
       ├── material
       ├── damage
       └── deckingCondition
```

### 6.4 Design Principle

The entity model is **fact-oriented**, not class-oriented. Every entity is defined by the `PolarisFact` records that attach to it. New entity types are added by defining their expected variables, not by writing new classes. A `tree_group` entity is just a label that facts reference via `entityType + entityId`.

## 7. Qualification Completeness

### 7.1 Calculation

```
qualificationScore = collectedCustomerFacts / totalRequiredFacts
```

Where:
- `collectedCustomerFacts` = count of facts with `status === 'collected'` AND `evidence.speaker === 'customer'` AND `normalizedValue !== null`
- `totalRequiredFacts` = count of required variables for the detected industry + service

### 7.2 Variable Classification

Each industry/service defines its variables as:

| Classification | Definition | Counts Toward Total? | Example |
|---|---|---|---|
| `required` | Must be collected for a complete qualification | Yes | tree height, service address |
| `optional` | Helpful but not required | No | species, disposal preference |
| `conditionally_required` | Required only if another variable is set | Yes, if condition met | stump removal required only if removal is requested |
| `informational` | Nice to know, no impact on qualification | No | customer preferred time |

### 7.3 What Does NOT Count

- Facts with `evidence.speaker === 'agent'`
- Facts with `status === 'mentioned_unresolved'`
- Facts with `status === 'conflicting'`
- Facts with `status === 'inferred'`
- Facts with `status === 'missing'`
- Facts with `normalizedValue === null`
- Keyword-only matches that produced no normalized value

### 7.4 Display

- `collected` with customer evidence → green checkmark
- `mentioned_unresolved` → amber circle with "Mentioned, not specified"
- `missing` → gray dash with "Not discussed"
- `conflicting` → amber warning with "Conflicting information"
- `inferred` → blue icon with "Inferred from [source]"
- `not_applicable` → hidden or grayed out

## 8. Estimate Eligibility

### 8.1 Which Facts May Affect the Estimate

| Fact Source | May Affect Range | May Affect Adjustments | May Affect Confidence | May Affect Range Width |
|---|---|---|---|---|
| Customer `collected` | Yes | Yes | Yes | Yes |
| Customer `mentioned_unresolved` | No | No | Yes (widens) | Yes |
| Customer `conflicting` | No | No | Yes (reduces) | Yes |
| System `inferred` | No | Only if explicitly permitted | Yes (reduces) | Yes |
| Agent-sourced | No | No | No | No |

### 8.2 Adjustment Composition

Current behavior: `Math.max()` selects one multiplier while retaining all reasons.

**New behavior:** Adjustments compose multiplicatively from only customer-sourced `collected` facts. Each adjustment records:
- The fact that triggered it (fact ID)
- The multiplier applied
- The reason text

```typescript
interface EstimateAdjustment {
  factId: string;
  variable: string;
  multiplier: number;
  reason: string;
}
```

### 8.3 Uncertainty and Range Width

- If any required variable is `missing` or `mentioned_unresolved`, the range width increases by a configurable uncertainty factor.
- If any `conflicting` facts exist, the range width increases further.
- Confidence is a separate dimension from range width. High confidence with wide range means "we're sure this is a broad estimate." Low confidence with narrow range means "we're guessing, but it's probably in this band."

### 8.4 Revenue Defaults

The current baseline ranges (`$1,800–$3,800` for Tree Service, etc.) remain untouched during Phase C. They become the fallback when no customer-sourced pricing facts exist.

## 9. Contact and Lead Actionability

### 9.1 Canonical Contact Record

```typescript
interface ContactRecord {
  name: PolarisFact | null;          // category or free_text
  phone: PolarisFact | null;         // phone — from dialed number, caller ID, or customer statement
  serviceAddress: PolarisFact | null; // address
  email: PolarisFact | null;         // email
  preferredContactMethod: PolarisFact | null; // category: 'phone' | 'email' | 'text' | 'any'
  appointmentPreference: PolarisFact | null;  // datetime or free_text
  isExistingCustomer: PolarisFact | null;     // boolean
  callOutcome: 'completed' | 'emergency_transfer' | 'refused' | 'disconnected' | 'voicemail' | 'no_answer';
}
```

### 9.2 Actionability Rules

| State | Definition | Can Schedule? | Can Send Pricing? | Can Call Back? |
|---|---|---|---|---|
| **Actionable** | Customer opted in, phone or address known | Yes | Yes | Yes |
| **Partially actionable** | Contact info known but no opt-in or appointment preference | Yes (with confirmation) | Yes | Yes |
| **Non-actionable** | No contact info, caller refused, or disconnected | No | No | No |
| **Emergency transfer** | Call was transferred to emergency services | N/A | N/A | N/A |

### 9.3 Refusal and Disconnect

- `refused` — Customer explicitly declined to provide information. Represented as a fact with `status: 'collected'`, `valueType: 'boolean'`, `normalizedValue: true`, `variable: 'refused_information'`.
- `disconnected` — Call ended unexpectedly. No inference about why. All in-progress facts become `mentioned_unresolved`.

## 10. Backward Compatibility

### 10.1 Coexistence Strategy

The new `PolarisFact[]` array lives alongside the existing `polarisIntelligence` record. The existing flat fields are populated from the facts array during a transitional period.

```typescript
// New — added alongside existing fields
polarisFacts: PolarisFact[];

// Existing — kept but populated FROM polarisFacts
customerFacts: { ... };           // derived from polarisFacts where entityType === 'customer'
estimatingVariables: [];           // derived from polarisFacts where status === 'collected'
missingInformation: [];           // derived from polarisFacts where status === 'missing' or 'mentioned_unresolved'
reasoning: [];                    // derived from polarisFacts provenance strings
executiveBriefing: string;        // unchanged — but generated from facts, not keywords
```

### 10.2 Compatibility Layer

A compatibility function `buildPolarisLegacyFromFacts(facts: PolarisFact[]): LegacyRecord` produces the old shape from the new model. This runs during the transitional period so existing consumers (dashboard, lead detail, report) continue working without changes.

### 10.3 What Is NOT Removed During Migration

- `polarisIntelligence` — kept until Phase F
- `estimatingVariables` — kept until Phase F
- `missingInformation` — kept until Phase F
- `reasoning` — kept until Phase F
- `executiveBriefing` — kept indefinitely (free text is useful)
- `polarisEstimate` flat fields — kept until Phase F
- Any consumer in `demo.js` or `public/index.html` — unchanged until frontend migration

### 10.4 Removal Criteria

Compatibility fields are removed only when:
1. All consumers have been migrated to the new model
2. Regression tests pass with the new model as primary
3. A release cycle has passed with no rollback

## 11. Phase C–F Migration Sequence

### Phase C — Evidence-Safe Extraction

**Scope:** Transcript normalization, speaker isolation, typed fact production

Steps:
1. Add `TranscriptTurn` normalization for Retell and simulation formats
2. Add `PolarisFact` schema to the codebase (no extraction logic yet)
3. Add compatibility layer (`buildPolarisLegacyFromFacts`)
4. Rewrite `extractValues()` to accept `TranscriptTurn[]` and filter by speaker
5. Rewrite `detectService()` to use only customer turns
6. Rewrite `analyzeTranscriptQualification()` to produce `PolarisFact[]`
7. Rewrite `extractSourceQuote()` to use turn boundaries, not character windows
8. Wire `buildPolarisIntelligence()` to consume `PolarisFact[]` and produce legacy fields via compatibility layer
9. Remove agent-sourced urgency, difficulty, and service detection

**Stop gate:** All existing tests pass. No regression in report rendering. Duration and recommendation fixes remain intact.

**Rollback:** Restore `buildPolarisIntelligence()` to its pre-Phase-C implementation. The compatibility layer is forward-only; the old code path is preserved in a backup branch.

### Phase D — Entity Model

**Scope:** Tree entity model, measurement/entity association, estimate safeguards

Steps:
1. Define `TreeGroup` and `Tree` entity types
2. Add entity association to `extractValues()` — associate measurements with the correct tree/group
3. Update `buildPolarisIntelligence()` to produce per-tree facts
4. Update estimate logic to compose adjustments from customer-sourced tree facts
5. Add range-widening for missing/unresolved facts
6. Add `conflicting` fact detection

**Stop gate:** Tree Service demo produces correct per-tree facts. Single-tree and multi-tree scenarios work.

**Rollback:** Remove entity model, fall back to flat per-job facts.

### Phase E — Contact Actionability

**Scope:** Canonical contact record, delayed `call_analyzed` enrichment, lead reconciliation

Steps:
1. Fix `callId` bug in `generateAndAttachSummary()` (webhook.js:243)
2. Populate `ContactRecord` from dialed number, caller ID, and customer-stated contact info
3. Populate `customerFacts` from saved lead data, not just `executiveSummary`
4. Extend SSE polling to continue past `completed` and capture `call_analyzed` enrichment
5. Reconcile duplicate leads from `call_ended` and `call_analyzed`

**Stop gate:** Contact data survives the full call lifecycle. `call_analyzed` enriches rather than duplicates.

**Rollback:** Revert SSE polling window, keep lead de-duplication fallback.

### Phase F — Next Steps Generator

**Scope:** Backend-ranked actionable step generator

Steps:
1. Build a ranked generator that checks contact availability, safety unknowns, unresolved facts, confidence, and appointment state
2. Remove hardcoded "Schedule on-site evaluation" / "Send follow-up with pricing" fallback
3. Generate conditional steps: "Confirm address" only if address is `missing`, "Call back" only if phone is available
4. Wire into post-call report

**Stop gate:** Generated steps are always correct for the current fact state. No steps are recommended when the lead is non-actionable.

**Rollback:** Revert to hardcoded fallback.

## 12. Test Strategy

### 12.1 Permanent Fixtures

Each fixture is a JSON file containing:
- Raw transcript turns (Retell or simulation format)
- Expected `PolarisFact[]` output
- Expected qualification score
- Expected estimate range
- Expected next steps

### 12.2 Fixture Catalog

| # | Scenario | Key Verification |
|---|---|---|
| F1 | Mixed customer/agent speech — Tree Service | Only customer turns produce `collected` facts. Agent urgency questions produce no evidence. |
| F2 | Agent urgency question ("Is this an emergency?") | Urgency fact is `inferred` or `missing`, not `collected`. No urgency score from agent speech. |
| F3 | "Four 120-foot trees" | Quantity=4 (`collected`), Height=120 (`collected`). Associated with tree_group. |
| F4 | Multiple trees, different sizes | Two tree facts: Tree A height=80, Tree B height=120. Each has distinct evidence. |
| F5 | Multiple measurements in one utterance ("80 feet tall, 2 feet diameter") | Two facts produced from one utterance. Both `collected`. |
| F6 | Conflicting customer statements ("80 feet" then "no, 100 feet") | Two facts, both `collected`, each with `conflicts: [otherId]`. |
| F7 | Unresolved mentions ("It's tall, maybe 50 feet?") | Height is `mentioned_unresolved` (low `extractionConfidence`, approximate). |
| F8 | Missing contact information | ContactRecord: phone=null, address=null. Lead is non-actionable. |
| F9 | Customer refuses to provide information | `refused_information: true`. All subsequent facts `mentioned_unresolved`. |
| F10 | Emergency transfer | `callOutcome: 'emergency_transfer'`. No estimate generated. |
| F11 | Disconnected call | Late facts become `mentioned_unresolved`. ContactRecord uses dialed number only. |
| F12 | Delayed `call_analyzed` | SSE continues polling until `call_analyzed` arrives. Facts enriched, not duplicated. |
| F13 | Simulation vs Retell format | Both normalize to same `TranscriptTurn` format. Same extraction produces same facts. |
| F14 | Production-shaped Tree Service (redacted) | The exact conversation that produced the original symptom cluster. Permanent regression fixture. |

## 13. Open Decisions Requiring Owner Approval

### 13.1 Entity Granularity — Tree Group vs Individual Tree **RESOLVED (Decision 2)**

The spec defines `treeGroups[]` with `trees[]` inside. An alternative is a flat `trees[]` with no grouping — each tree is independent.

**Owner decision:** Use `treeGroups[]` with nested `trees[]`. Supports one tree, multiple similar trees as a group, and multiple trees with different facts. Do not force artificial individual records when the customer only provides grouped information.

### 13.2 Conflict Resolution **RESOLVED (Decision 3)**

When the customer says "80 feet" then "no, I meant 100 feet", the current spec produces two `collected` facts with `conflicts: [otherId]`. The UI shows both.

**Owner decision:** Keep both facts. Preserve every conflicting customer-supported value. Latest statement is identified as most recent candidate but does not automatically replace earlier evidence unless the customer explicitly corrects the prior statement.

### 13.3 Inferred Fact Permissions for Estimation **RESOLVED (Decision 4)**

The spec says inferred facts may affect estimate range width and confidence, but not the base multiplier. An alternative is to allow inferred facts to apply a reduced multiplier.

**Owner decision:** No multiplier from inferred facts. Only widen the range and lower estimate confidence. This prevents garbage-in/garbage-out pricing.

### 13.4 Fixture Storage **RESOLVED (Decision 5)**

**Owner decision:** `tests/fixtures/polaris/` with subdirectories for transcripts, expected-facts, expected-intelligence, retell-payloads, and simulation-payloads. The failed production-shaped Tree Service scenario must be included as a permanent regression fixture.

### 13.5 Verification Confidence

The owner has decided NOT to add `verificationConfidence` yet. It may be introduced only when NorthStar has a real verification mechanism (repeated customer confirmation, cross-source agreement, business records, property data, completed-job outcomes, or human review). This remains open as a future consideration.

## Summary

This specification defines a typed, provenance-carrying fact model that:

1. **Preserves evidence** — speaker, turn, utterance, exact span
2. **Enforces boundaries** — customer speech is not agent speech, agent questions are not customer statements
3. **Supports entities** — tree groups, HVAC systems, rooms, roof sections
4. **Defines invariants** — only `collected` + customer-sourced counts toward qualification and pricing
5. **Coexists safely** — compatibility layer keeps existing consumers working
6. **Stages migration** — Phases C–F in dependency order with stop gates

The model is the smallest one that solves the confirmed defects without over-engineering for unvalidated future needs.
